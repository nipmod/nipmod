# Strategic Decision Register

This register records accepted strategic decisions so Nipmod does not drift every time the market narrative changes.

## D-001: Make API-first the public product

Status: accepted

Nipmod's primary public product is one package intelligence API for agents. The website, README, posts and docs should lead with the hosted API. CLI, MCP and setup paths are support surfaces, not the headline.

Reason:
Agents do not need a different native integration for every host. If an agent can call HTTPS or MCP, it can use Nipmod. This makes distribution broader and less dependent on platform partnerships.

## D-002: Remove Gitlawb from the foundation

Status: accepted

Gitlawb remains a supported source and a possible source provenance path, but it is no longer the foundation of the company story.

Reason:
Nipmod should not be judged by Gitlawb's team speed, product pivots or public credibility. GitHub and the Nipmod API are the stable public foundation.

## D-003: Keep GitHub as the main credibility surface

Status: accepted

GitHub is the primary public repo, CI surface, issue surface, documentation surface and review surface.

Reason:
Developers, investors, security reviewers and external builders understand GitHub immediately. It gives Nipmod a stronger default trust signal than a smaller source network.

## D-004: Keep platform integrations secondary

Status: accepted

Codex, Claude Code, Cursor, Hermes, OpenCode and similar surfaces should be described as ways agents can use Nipmod, not as the core product.

Reason:
Platform-by-platform integration marketing creates dependency and slows adoption. The stronger message is that any agent with HTTPS or MCP can use the same API.

## D-005: Archive only useful confirmed records

Status: accepted

Nipmod should not bulk mirror public registries into its own archive. External package records are resolved on demand. Durable archive records should require confirmation, qualification or owner claim.

Reason:
Raw volume creates spam, legal risk and low trust. Curated memory creates a defensible package intelligence asset.

## D-006: Make trust ranking the core moat

Status: accepted

The strategic technical priority is better package decisions: source reliability, project health, maintainership, license, security, install risk, popularity, recency, fit and historical confirmations.

Reason:
Agents can already search the web. Nipmod wins only if it makes better, safer and more reusable package decisions.

## D-007: Keep $NPM visible but not dominant

Status: accepted

$NPM can be linked and explained in roadmap contexts, but product pages and technical docs should lead with the API and trust system.

Reason:
Developer credibility is damaged if the token appears to be the product. The token should follow platform usage, not replace it.

## D-008: Separate public docs from historical launch docs

Status: accepted

Old Gitlawb-first docs, launch packets and integration claims should either be rewritten, archived under legacy context or clearly marked as historical.

Reason:
Public docs should not contradict the current API-first direction.

## D-009: Preserve the current visual design

Status: accepted

The current website design is kept. Strategy cleanup should change copy, hierarchy and navigation only when needed. No redesign unless explicitly requested.

Reason:
Design churn slows product work. The immediate problem is strategic clarity, not visual identity.

## D-010: Prioritize source resolver and trust engine depth

Status: accepted

The next technical expansion should focus on resolver depth, trust scoring, install-plan quality and archive confirmation logic.

Reason:
This compounds the core product. More logos, platform pages or unreviewed integration claims do not.
